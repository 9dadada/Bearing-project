================================================================
발표 자료 생성용 프롬프트  (CWRU 베어링 이상탐지 — 비지도 오토인코더 담당)
이 텍스트 전체를 AI(ChatGPT/Claude 등)에 붙여넣어 발표 슬라이드 구성을 생성하세요.
================================================================

당신은 학부 캡스톤 발표 슬라이드를 설계하는 전문가입니다. 아래 프로젝트 내용으로
"슬라이드별 (제목 / 핵심 메시지 한 줄 / 들어갈 그림·표 / 30초 발표 스크립트)"를 표로 만들어 주세요.
발표자는 "비지도학습 기반 정상/이상 분류(오토인코더, 경로 B)"를 담당했습니다.
그 부분을 중심으로 전체 시스템 맥락도 함께 설명합니다. 한국어, 공학 발표 톤, 수치는 그대로 사용.

────────────────────────────────────────────────────────────
[1] 프로젝트 개요
────────────────────────────────────────────────────────────
- 주제: CWRU 베어링 진동 신호 기반 고장 이상탐지 시스템
- 데이터: CWRU 베어링 데이터셋(.mat, 가속도 진동). 전 신호를 12kHz로 통일(48k는 다운샘플) → 단일 파이프라인
- 3경로 게이트형 파이프라인:
  · 경로 A: 통계 특징(RMS/peak/첨도/파고율)+임계값 — 최종적으로 미사용(버림). 버린 이유는 [5]에서 설명
  · 경로 B: 디노이징 Conv 오토인코더(비지도) — 정상/이상 탐지       ★발표자 담당
  · 경로 C: 물리 엔벨로프 진단(BPFO/BPFI/BSF) + CNN(MobileNetV2) 교차검증 — 고장유형(내륜/외륜/볼) 분류
- 게이트: 경로 B가 '이상'으로 판정 → 디바운스(연속 3윈도우) → 그때만 경로 C 호출
- 팀 역할: 경로 A/B/융합=발표자, 경로 C+백엔드=팀원1, 프론트엔드=팀원2

────────────────────────────────────────────────────────────
[2] ★파이프라인 데이터 흐름 (슬라이드 1장, 하나의 도식으로 가독성 높게)
────────────────────────────────────────────────────────────
원시 진동 신호(1D 시계열, 12kHz)에서 두 갈래로 분기 — 분기점이 그림에 명확히 보여야 함:

  원시 신호(1D 숫자) ──┬── (A) 전처리[윈도잉 1024pt≈2.5회전 + z-score 정규화]
                        │         → STFT → 시간-주파수 스펙트로그램(2D 이미지)
                        │              ├─→ ★경로 B: 디노이징 오토인코더 → 재구성오차 → 임계값 → 정상/이상
                        │              └─→  경로 C-CNN: MobileNetV2 → 고장유형(IR/OR/B)
                        │
                        └── (B) 원시 신호 그대로 → 경로 C-물리: 엔벨로프 분석 → 결함주파수(BPFO/BPFI/BSF) → 고장유형

  게이트: 경로 B가 '이상'일 때만 경로 C 실행 → 경로 C에서 물리 vs CNN 교차검증 → 최종 진단

  ※ 그림에 반드시 드러나야 할 3가지:
    (1) "숫자(1D) → 스펙트로그램(이미지) → 오토인코더"
    (2) "원시 숫자 데이터 → 경로 C 물리 진단"
    (3) "스펙트로그램 이미지 → 경로 C CNN"
    → 같은 스펙트로그램 이미지가 경로 B(오토인코더)와 경로 C(CNN) 양쪽에 쓰인다는 점이 핵심.

────────────────────────────────────────────────────────────
[3] ★발표자 모델 — 디노이징 Conv 오토인코더 (비지도학습)
────────────────────────────────────────────────────────────
- 학습 방식: 정상 데이터만 사용(비지도). 정상 스펙트로그램에 가우시안 노이즈(σ=0.1)를 더한 뒤
  '깨끗한 원본 전체'를 복원하도록 학습(디노이징). 고장 데이터는 학습에 일절 미사용.
- 추론: 새 신호의 재구성오차가 크면 이상. 임계값 = 정상 재구성오차의 평균 + 3σ.
- 모델 구조:
  · 입력: 1채널 스펙트로그램 (약 129×25; n_fft=256, hop=32)
  · 인코더: Conv2d 3단(stride2, padding1, ReLU), 채널 1→16→32→64  (공간 압축)
  · 병목: Flatten → Linear(→128) → Linear(→복원)  [잠재차원 128 = 강제 압축으로 '정상의 본질'만 학습]
  · 디코더: ConvTranspose2d 3단(stride2, ReLU), 채널 64→32→16→1  (복원)
  · 총 Conv 6개(인코더3 + 디코더3) + Linear 2개, 파라미터 약 116만
  · 학습: MSE 손실, Adam(lr=0.001), 30 epoch, batch 64

- (실험1) 일반 AE vs 디노이징 AE — '탐지 성능' 비교 (정직하게 보고할 것):
  · 노이즈 없는 환경 + 화이트/임펄스 노이즈에서 탐지 AUC·오탐률을 측정한 결과,
    일반 AE / 디노이징(σ=0.1) / 디노이징(σ=0.5)이 사실상 동일 (예: 임펄스 강 AUC 82.7% / 82.7% / 82.9%).
  · 결론: 디노이징이 탐지 성능을 더 좋게 하지는 못했다(학습 노이즈가 스펙트로그램 레벨이라
    신호레벨 테스트 노이즈에는 효과가 적음). 디노이징은 표준 정규화 기법으로 해는 없으며,
    핵심 성능은 '오토인코더 구조 자체'에서 나온다. (과장 금지 — 솔직한 비교가 신뢰를 줌)

- (실험2) 병목(잠재차원) 크기 ablation — production ConvAutoencoder, 32 vs 64 vs 128:
  ┌────────┬──────────┬──────┬────────┬───────────┬─────────────┐
  │ 병목    │ 파라미터  │ AUC  │ 오탐률 │ 부하0 오탐 │ 정밀도 99:1  │
  ├────────┼──────────┼──────┼────────┼───────────┼─────────────┤
  │  32     │  33만    │ 1.0  │ 0.49%  │ 3.4%      │ 68%         │
  │  64     │  61만    │ 1.0  │ 0.49%  │ 3.4%      │ 68%         │
  │ 128 ★   │ 117만    │ 1.0  │ 0.36%  │ 2.5%      │ 74%         │
  └────────┴──────────┴──────┴────────┴───────────┴─────────────┘
  · 탐지율·AUC는 셋 다 100%/1.0. 차이는 '오탐 마진'에서 남.
  · 병목↓ → 파라미터↓(경량)이지만 오탐률↑·정밀도↓ (압축이 세져 정상 복원 여유가 줄어듦).
  · 64는 32와 성능 동일한데 파라미터만 2배 → 열등(선택 가치 없음).
  · 결론(실험으로 도출): 경량화하려면 32까지 줄여도 탐지는 되지만 오탐 마진이 깎인다.
    정확도 우선이므로 → 병목 128 채택.

- (실험3) 레이어 수 ablation — Conv 단 수 2~5 (병목 128 고정):
  · 노이즈 없음 AUC: 전부 1.0 (레이어 수와 무관하게 완전 분리)
  · 임펄스 강 AUC : 2단 80.5% / 3단 82.7% / 4단 83.9% / 5단 83.3% → 깊을수록 강건성 약간↑, 4단 정점 후 포화
  · 노이즈 없음 오탐률 : 2·3·5단 0.36% / 4단 0.61%
  · 결론(실험으로 도출): 3단이 균형점 — 강건성 충분 + 오탐 최저 + 구조 단순. 4단 이득은 미미. → 레이어 3단 채택.

[4] ★오토인코더 신뢰도 평가 — 5개 지표 + 평가 과정
────────────────────────────────────────────────────────────
평가 과정: 정상만 학습 → 임계값(평균+3σ) 설정 → 정상/고장에 노이즈를 단계별로 주입하며
           AUC·탐지율·오탐률 측정 → base rate별 정밀도 계산 → 부하 hold-out으로 일반화 검증.

① 분리도   : 정상 이상점수 평균 0.86 vs 고장 5.46 (임계값 1.0 기준, 분포 겹침 0)
② 탐지율/오탐률 : 탐지율 100% / 오탐률 0.4%
③ 정밀도(현실 base rate별, '고장이라면 맞을 확률') : 50:50→100%, 90:10→97%, 95:5→94%, 99:1→74%
④ 노이즈 강건성(AUC, 화이트·임펄스 동일 SNR dB) :
     화이트 10/0/-10dB = 모두 100% (분별 정보 유지) / 임펄스 10/0/-10dB = 96/84/76%
     → 임계값이 흔들려도(오탐률 따로) 분별력(AUC)은 유지/완만 하락 = 강건
⑤ 부하별 오탐(전체 부하 0~3 학습) : 부하0 2.5%, 부하1·2·3 0%
참고: 노이즈 없는 데이터 AUC = 1.0 (가장 작은 결함 0.007"도 정상과 완전 분리, 겹침 0)
한계(정직하게) : 학습하지 않은 운전조건(부하 hold-out)에선 오탐 100% = 재구성기반 이상탐지의 도메인 의존성
     → 대응: 배포 시 모든 운전조건의 정상 데이터를 학습에 포함 / fs·rpm 입력받아 정렬(order tracking)

────────────────────────────────────────────────────────────
[5] ★왜 오토인코더인가 — 통계 도구(경로 A)가 산업 현장에서 못 쓰이는 이유 (강건성)
────────────────────────────────────────────────────────────
- 경로 A(RMS/peak/첨도/파고율 + 임계값)는 '진폭 스파이크'에 즉각 반응 →
  기계 소음·충격·전기 노이즈에 쉽게 오탐.
- 실제 사례: 정상 신호(normal_0)에서 경로 A가 연속 3윈도우 오탐 → 전체를 고장으로 오판
  (반면 같은 구간에서 오토인코더는 정상으로 정확 판정). → 이것이 경로 A를 버린 직접 근거.
- 노이즈 강건성 비교(AUC): 임펄스 노이즈에서 경로 A의 AUC는 ~0.66까지 급락,
  경로 B(오토인코더)는 0.8 이상 유지. (탐지율은 둘 다 100%지만 분별력=AUC에서 갈림)
- 산업 현장 = 부하 변동·기계 충격·배경 진동 상존 → 통계 임계값은 이런 노이즈를 고장으로 착각해
  오탐 폭증 → '양치기 소년'이 되어 실사용 불가. 그래서 경로 A를 버리고 오토인코더 단독으로 결정.
- 오토인코더는 '정상의 시간-주파수 패턴' 자체를 학습 → 노이즈에 강건, 임계값 무관 분별력(AUC) 우수.

────────────────────────────────────────────────────────────
[6] ★데이터 누수 방지 설계 — "AUC 1.0이 부풀려진 게 아니다" 방어
────────────────────────────────────────────────────────────
- 오토인코더는 정상 데이터만으로 학습(비지도). 고장 데이터는 학습·임계값 설정에 일절 미사용.
  → 평가 때 고장은 모델이 '처음 보는 데이터'. AUC 1.0은 본 적 있는 걸 맞춘 게 아님.
- recording(녹음) 단위 분리: 같은 녹음의 윈도우가 학습/평가에 뒤섞이지 않음 → 윈도우 누수 차단.
- 임계값도 정상 재구성오차로만 설정 → 고장 정보가 임계값에 새지 않음.
- 따라서 AUC 1.0은 'CWRU 결함이 워낙 뚜렷(0.007"도 완전 분리)'해서 나온 정직한 결과이지,
  데이터 누수로 부풀린 수치가 아니다. (오히려 더 어려운 검증은 노이즈·운전조건 — [4]④⑤)

────────────────────────────────────────────────────────────
[7] ★게이트 구조의 효율성 — 실시간성
────────────────────────────────────────────────────────────
- 경로 B(가벼운 오토인코더)가 '정상'으로 판정하면 무거운 경로 C(CNN MobileNetV2 + 물리 엔벨로프 분석)를
  호출하지 않는다.
- 현장은 대부분의 시간이 '정상' → 대부분의 입력은 가벼운 경로 B만 통과 → 연산 비용 절약.
- 무거운 딥러닝 분류·물리 진단은 '이상일 때만' 수행 → 실시간 상시 모니터링에 적합.
- 디바운스(연속 3윈도우 이상일 때만 알람)로 순간 오탐을 걸러 불필요한 경로 C 호출도 억제.

────────────────────────────────────────────────────────────
[8] 발표 흐름 제안 + 요청사항
────────────────────────────────────────────────────────────
흐름: 문제정의 → 데이터/전처리 → 파이프라인 한눈에(분기 도식) → 왜 통계도구는 안 되는가(강건성) →
      내 모델(디노이징 오토인코더 구조·학습) → 신뢰도 평가 5지표 → 누수방지·게이트효율 →
      한계·향후과제 → (라이브 데모) → 역할분담·결론

요청:
- 슬라이드별로 (제목 / 핵심 한 줄 / 들어갈 그림·표 / 30초 발표 스크립트)를 표로 정리
- [2]의 파이프라인 분기 도식을 구체적인 도형 배치(박스·화살표)로 제안
- 위 수치를 그대로 사용하고, 한계와 '디노이징이 더 낫지는 않았다'는 점도 정직하게 포함
- 발표자(오토인코더) 슬라이드에 가장 많은 비중을 둘 것

────────────────────────────────────────────────────────────
[9] ★PPT용 코드 설명 (발표자 코드 슬라이드)
────────────────────────────────────────────────────────────
발표에서 보여줄 핵심 코드 3덩어리 — 각각 "무슨 일을 하는지" 한 줄 요지와 함께 슬라이드화.

(1) 모델 정의 — src/autoencoder_detector.py : ConvAutoencoder
    · 인코더 nn.Conv2d ×3 (stride2, 채널 1→16→32→64) → Flatten → Linear(→128) 병목
    · 디코더 Linear(→복원) → nn.ConvTranspose2d ×3 (64→32→16→1)
    · 요지: "병목 128로 강제 압축 → 정상의 핵심만 학습 → 정상을 못 복원하면(재구성오차 큼) 이상"

(2) 학습 — train_autoencoder.py (정상만, 비지도)
    · 정상 신호 → 윈도잉 → STFT 스펙트로그램 → 정규화(정상 통계만, 저장)
    · train_ae(): 입력에 가우시안 노이즈 더해 깨끗한 원본 복원(디노이징), MSE 손실, 30 epoch
    · 임계값 = 정상 재구성오차 평균 + 3σ → thresholds.json 저장
    · 요지: "고장은 학습에 일절 안 씀 = 데이터 누수 차단([6]과 연결)"

(3) 추론·게이트 — src/inference.py : run_recording()
    · ab_scores() → fuse_stream(rule='b_only', 디바운스 연속3) → is_anomaly
    · 이상일 때만 경로 C 호출:   if is_anomaly: pathc = diagnose_recording(...)
    · 요지: "오토인코더가 이상이라 해야만 무거운 C 실행 = 게이트로 연산 절약([7]과 연결)"

요청: 위 3덩어리를 각각 (코드 5~10줄 발췌 + 화살표 주석 + 한 줄 요지)로 슬라이드화.
      실제 코드 라인은 발표 때 IDE에서 직접 보여주는 방식 권장.
